1. CentroCostosOpn.js (Presentación / Lógica Frontend)

Este archivo controla la interacción del usuario con la interfaz web, manejando eventos como clics en botones, validaciones de formularios, envíos de datos y actualizaciones dinámicas de la pantalla. Aquí normalmente se ejecutan las funciones CRUD desde el lado cliente, realizando llamadas hacia la capa de aplicación mediante requests o formularios. Su propósito es brindar una experiencia funcional al usuario sin modificar directamente la lógica de negocio. Cambiar este archivo afecta principalmente comportamiento visual o interacción, pero no altera reglas del sistema.

2. CentroCostosOpn.jsp (Presentación / Vista Web)

Este archivo representa la estructura visual de la pantalla: formularios, tablas, botones, campos de captura y componentes HTML/JSP necesarios para mostrar y capturar información del catálogo de centros de costos. Es la capa de presentación renderizada en el servidor, donde se conecta el frontend con los datos provenientes de la aplicación. Mientras el .js controla comportamiento, el .jsp define diseño y estructura de la interfaz.

3. CreCatCentroCostosOpnApp.java (Aplicación / Interfaz)

Esta interfaz define los métodos disponibles para la capa de aplicación, como guardar, consultar, actualizar o eliminar registros. Actúa como contrato formal entre presentación y aplicación, estableciendo qué operaciones puede consumir el sistema sin depender directamente de implementaciones concretas. Esto permite desacoplamiento y facilita mantenimiento.

4. CreCatCentroCostosOpnAppImpl.java (Aplicación / Implementación)

Aquí se implementan los métodos definidos en la interfaz de aplicación. Su función principal es recibir solicitudes desde la interfaz (JSP/JS), coordinar llamadas a la capa de negocio y devolver resultados. No suele contener lógica compleja, sino actuar como intermediario estructurado entre presentación y negocio.

5. CreCatCentroCostosOpnB1.java (Negocio / Interfaz)

Esta interfaz establece las operaciones de negocio relacionadas con centros de costos, definiendo procesos empresariales como validaciones, reglas o procesos CRUD. Es el contrato que permite separar lógica empresarial de sus implementaciones.

6. CreCatCentroCostosOpnB1Impl.java (Negocio / Implementación)

Este archivo contiene la lógica empresarial real del módulo. Aquí se ejecutan validaciones, reglas de negocio, restricciones y coordinación de operaciones antes de llegar a la base de datos. Es una de las capas más importantes porque controla que las acciones cumplan políticas del sistema.

7. CreCatCentroCostosOpnDAO.java (Persistencia / Interfaz DAO)

Esta interfaz define las operaciones de acceso a datos, como insertar, consultar, actualizar y eliminar registros en la base de datos. Es el contrato de persistencia que abstrae cómo se manipulan realmente los datos.

8. CreCatCentroCostosOpn.java (Entity / POJO Persistente)

Esta clase representa la tabla CRE_CAT_CENTRO_COSTOS_OPM como objeto Java. Cada atributo corresponde a una columna de la tabla y sirve como estructura de transporte de datos entre capas. Hibernate utiliza esta clase para mapear información entre Java y SQL.

9. CreCatCentroCostosOpnHibernateDao (Persistencia / Implementación DAO)

Esta clase implementa las operaciones definidas en el DAO usando Hibernate. Aquí se ejecutan realmente las consultas, inserciones, actualizaciones y eliminaciones sobre la base de datos mediante sesiones Hibernate. Es la capa final que conecta el sistema con la información persistente, traduciendo objetos Java en operaciones SQL reales.

CrearCentroCostosOpn.hbm.xml (Persistencia / Mapeo POJO ↔ Tabla)

Este archivo define cómo Hibernate relaciona tu clase Java (POJO) con la tabla real en la base de datos (CRE_CAT_CENTRO_COSTOS_OPM). Aquí se especifican los nombres de columnas, tipos de datos, claves primarias y atributos de la clase. Básicamente actúa como traductor entre objetos Java y registros SQL, permitiendo que cuando guardes, consultes o actualices un objeto, Hibernate sepa exactamente en qué tabla y columnas trabajar sin escribir SQL manual. Es fundamental para que el ORM funcione correctamente.

hibernate.cfg.xml (Persistencia / Configuración Hibernate)

Este archivo contiene la configuración general de Hibernate, como conexión a la base de datos, dialecto SQL, credenciales, pool de conexiones y registros de mapeos (.hbm.xml). Es el núcleo técnico que le dice a Hibernate cómo conectarse y operar. Si este archivo está mal configurado, el sistema no podrá iniciar la persistencia. En entornos legacy suele integrarse con Spring, pero sigue siendo la base de la comunicación ORM con la BD.

PersistenceContext.xml (Persistencia / Mapeo de Implementaciones DAO)

Aquí se registran los beans de la capa de persistencia, normalmente DAOs o repositorios. Su función es decirle a Spring qué clases implementan las operaciones de acceso a datos y cómo inyectarlas en otras capas. Por ejemplo, si existe CentroCostosDaoImpl, este archivo lo declara para que Spring pueda usarlo automáticamente. Es el puente organizacional entre Hibernate y la lógica empresarial, asegurando que la capa de negocio pueda consumir la persistencia sin crear objetos manualmente.

BusinessContext.xml (Negocio / Lógica de Servicios)

Este archivo configura los servicios de negocio (Business Services), donde reside la lógica empresarial real: validaciones, reglas, procesos y coordinación entre persistencia y aplicación. Aquí se definen clases como CentroCostosBusinessImpl, que reciben datos desde la aplicación, ejecutan reglas y llaman a los DAOs correspondientes. Si tienes dudas aquí, probablemente sea porque necesitas registrar correctamente la implementación y sus dependencias. Sin este contexto, la lógica del CRUD puede existir en código pero no ser reconocida por Spring.

ApplicationContext.xml (Aplicación / Fachada o Capa de Exposición)

Este archivo registra la capa más externa, normalmente servicios de aplicación o facades que conectan frontend/controladores con la lógica de negocio. Aquí se exponen implementaciones como CentroCostosAppServiceImpl, que reciben solicitudes del sistema (pantallas, controladores o JS), delegan a negocio y devuelven resultados. En una arquitectura monolítica empresarial, esta capa organiza el flujo completo y desacopla presentación de negocio. Es esencial para que el sistema pueda invocar correctamente tu módulo CRUD desde la interfaz o controladores.
